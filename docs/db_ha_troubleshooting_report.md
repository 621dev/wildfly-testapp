# DB HA & Replication 모니터링 시스템 구축 및 장애 조치 완료 보고서

## 1. `test.jsp` 트러블슈팅 및 기능 정상화

### 1.1 데이터 저장 및 삭제 불가 장애
* **현상**: 이름 등록 폼 제출 및 쓰레기통(삭제) 아이콘 클릭 시, 백엔드 DB에 아무런 트랜잭션이 발생하지 않고 화면 리다이렉트 및 오동작이 지속됨.
* **원인 분석**:
  1. **Multipart 요청 파싱 불가**: 프론트엔드 자바스크립트(`fetch`)에서 `new FormData()` 객체를 페이로드로 직접 보냄에 따라 헤더가 `multipart/form-data`로 인코딩되었습니다. 별도의 멀티파트 리졸버가 없는 표준 JSP의 `request.getParameter()`는 이를 읽지 못해 모든 파라미터가 `null`이 되었습니다.
  2. **JSP EL 엔진과 JS 백틱 충돌**: 동적 HTML 생성 백틱(`` ` ``) 템플릿 내의 JS 변수 바인딩 구문(`${data.id}`, `${escapeHtml(data.name)}`)을 JSP 컴파일러가 **JSP EL(Expression Language) 표현식**으로 오인하여 컴파일 오류 및 스크립트 블록 전체 로드 실패(`deleteRow is not defined` 에러)를 유발했습니다.
* **조치 사항**:
  1. **데이터 직렬화 포맷팅 변경**: `FormData`를 브라우저 표준인 **`URLSearchParams`**로 래핑하여 일반 폼 전송 방식(`application/x-www-form-urlencoded`)으로 전송을 변경하여 `request.getParameter()`가 완벽히 인식하도록 조치했습니다.
  2. **JSP EL 비활성화**: JSP 최상단 페이지 디렉티브에 `isELIgnored="true"` 옵션을 추가하여 컴파일러가 JS ES6 템플릿 구문을 무시하고 브라우저로 날것 그대로 넘기도록 차단했습니다.
  3. 한글 깨짐 방지를 위해 POST 요청 시 `request.setCharacterEncoding("UTF-8");`을 수립했습니다.

## 2. DB HA 모니터링 고도화 (VIP 환경 대응)

### 2.1 가상 IP(VIP) 환경에서의 헬스체크 한계 극복
* **기존 한계**: 두 DB 노드가 VIP(`10.10.20.21`)로 묶인 경우, WAS의 데이터소스가 VIP를 바라보므로 `setReadOnly(true)` 핑 테스트를 날려도 L4/이중화 스위칭에 의해 마스터 노드로만 트래픽이 우회되어 슬레이브 노드의 실제 물리 장애 여부를 감지하지 못했습니다.
* **고도화 조치**:
  1. **마스터 커넥션 기반 복제 실시간 감시**: WAS가 물리 슬레이브 서버에 직접 붙는 대신, 마스터 연결을 통해 **`SHOW SLAVE HOSTS`**를 쿼리하여 복제에 연결되어 있는 하위 슬레이브 노드 목록과 생존 상태를 실시간 수집하는 우아한 방식으로 변경했습니다.
  2. **실체적 물리 IP 도출 로직 추가**: VIP 뒤에 가려진 진짜 접속 서버를 색출하기 위해 데이터베이스의 고유 식별값인 **`@@server_id`** 및 호스트네임(`@@hostname`)을 즉석 조회하여, 마스터(`10.10.20.4`)와 슬레이브(`10.10.20.5`)의 **진짜 물리 IP를 판별해 화면에 2줄로 명확하게 뿌려주는 지능형 매핑 코드**를 추가했습니다.

## 3. Jenkins CI/CD 자동 배포 구축

수동으로 진행하던 빌드 및 배포 과정을 깃 push 시점에 맞춰 젠킨스가 스스로 감지하도록 `Jenkinsfile`을 고도화했습니다.

```groovy
    // Jenkinsfile 내 triggers 블록 내장
    triggers {
        // 5분마다 Git 저장소 폴링 감시
        pollSCM('H/5 * * * *')
        // GitHub 웹훅 알림 실시간 트리거 활성화
        githubPush()
    }
```
* **효과**: 개발자가 코드를 수정하여 `git push`만 실행하면 최대 5분 이내에 젠킨스가 이를 인지하고 자동으로 빌드, 롤링 배포, 헬스체크 단계까지 무인 실행하게 되었습니다.

## 4. VIP 이중화(Keepalived) Failover & Failback 장애 조치

마스터 다운 후 복구하는 과정에서 발생한 가상 IP(VIP) 꼬임 및 통신 불가 상태에 대해 심층 물리 디버깅을 진행하여 완벽히 수리했습니다.

### 4.1 Failover 시의 Connection Refused 및 ARP 캐시 꼬임
* **현상**: 마스터 DB를 중단시켜 VIP가 슬레이브 노드로 성공적으로 이전(`ip addr show`로 확인)되었음에도 불구하고, WAS 단에서 DB 접속이 완전히 불통되는 현상이 일어났습니다.
* **원인**: **ARP 캐시 브로드캐스트 미동기화**. VIP는 슬레이브로 이전되었으나 WAS 서버의 메모리상 ARP 테이블에는 가상 IP `10.10.20.21`에 매핑된 MAC 주소가 옛날 마스터 장비 주소로 그대로 고정되어 있어 패킷이 엉뚱한 곳으로 날아갔습니다.
* **조치**: 슬레이브 DB 노드에서 Gratuitous ARP 패킷을 강제로 방출하여 네트워크 스위치와 WAS의 캐시를 즉시 동기화했습니다.
  ```bash
  # 슬레이브 DB에서 강제 실행
  arping -U -c 5 -I ens36 10.10.20.21
  ```

### 4.2 마스터 복구 시의 VIP 중복 기상 (Split-Brain)
* **현상**: 마스터 DB 서비스를 다시 켰을 때, 슬레이브의 VIP가 내려가지 않고 마스터와 슬레이브 두 물리 서버 모두에 동일한 VIP가 동시에 기상하여 네트워크 충돌이 일어났습니다.
* **원인**: **가상화(VMware) 환경의 Multicast 차단**. 두 서버의 Keepalived가 서로 생존 신호(VRRP 패킷)를 주고받을 때 멀티캐스트 방식을 쓰는데, VMware 가상 스위치와 OS 방화벽이 이 패킷을 드롭시켰습니다. 이에 따라 마스터와 슬레이브는 서로의 존재를 모른 채 둘 다 Active 마스터 권한을 선언했습니다.
* **조치**:
  1. **방화벽 프로토콜 개방**: 양쪽 OS 방화벽에 VRRP 직접 통과 룰을 심었습니다.
     ```bash
     firewall-cmd --direct --add-rule ipv4 filter INPUT 0 -p vrrp -j ACCEPT
     firewall-cmd --reload
     ```
  2. **유니캐스트(Unicast) 전환**: 멀티캐스트 차단 한계를 완벽히 회피하기 위해, 일대일 직접 대상 IP를 타겟 지정하는 유니캐스트 프로토콜 설정을 킵얼라이브 설정에 이식했습니다.
     ```ini
     # /etc/keepalived/keepalived.conf 내 unicast 지정
     unicast_src_ip [자신의_물리_IP]
     unicast_peer {
         [상대방_물리_IP]
     }
     ```
     > [!TIP]
     > 이 설정 적용 즉시 서로의 생사를 완벽히 인지하여, 마스터 기상 시 슬레이브의 VIP가 즉시 내려가고 마스터로 안전하고 확실하게 VIP가 자동 회수(Auto-Failback)되는 최적의 이중화 통신이 수립되었습니다.

## 5. 고가용성(HA) 아키텍처 의사결정 가이드

복구(Failback) 시의 데이터 동기화와 관련하여 미래 지향적인 아키텍처적 조언을 담았습니다.

### 5.1 복구(Failback) 시의 트랜잭션 수동 정합성 검증의 필요성
자동 Failback(원복) 시 슬레이브가 독립 기동했던 시간 동안 쌓인 신규 데이터(트랜잭션)가 마스터로 자동으로 역방향 동기화되지 않으므로, **수동 Failback 모드(비선점형)**를 유지하며 엔지니어가 데이터의 정합성을 수동으로 맞춰주는 것이 엔터프라이즈 환경의 가장 안전한 신성한 정석입니다.

### 5.2 DB 2대 환경에서의 "Galera Cluster" 구성 전략 (`Garbd` 활용)
2대의 고사양 DB 노드만으로 완전 동기식 Galera Cluster를 사용하게 되면, 네트워크 단절이나 한 노드 사망 시 과반수(Quorum) 불충족으로 인해 남은 한 대마저 스스로 동작을 멈추는 **동반 자살(Non-Primary) 현상**이 일어납니다.

이러한 리스크를 비용 추가 없이 완벽하게 돌파하는 해법은 **`Garbd` (Galera Arbitrator, 중재자)** 프로세스를 활용하는 것입니다.

```
       [ WAS 서버 (10.10.20.2) ]
       ┌────────────────────────┐
       │   Garbd (중재자) 🗳️     │  ← 데이터 저장 없이 오직 투표권 1표만 제공!
       └───────────┬────────────┘  
                   │
         ┌─────────┴─────────┐
         │                   │
┌────────┴────────┐ ┌────────┴────────┐
│ DB 1 (10.10.20.4)│ │ DB 2 (10.10.20.5)│
└─────────────────┘ └─────────────────┘
```

* **권장 아키텍처**:
  1. 실제 고사양 DB 서버 2대에 Galera Cluster를 설치합니다.
  2. WAS 서버(`10.10.20.2`) 구석에 데이터를 저장하지 않고 투표권 1표만 지니는 매우 가벼운 **`Garbd`** 데몬을 띄워 둡니다.
  3. **효과**: 총 3표의 의결권이 형성되어, DB 1번이 죽더라도 살아남은 DB 2번은 본인(1표) + Garbd(1표) = 2표(과반수 66%)를 즉시 획득하므로, 락이 걸리지 않고 단독으로 정상 서비스를 무중단 지속하게 됩니다!
