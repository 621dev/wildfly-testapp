# MariaDB Galera Cluster vs GTID 기반 비동기 복제 비교 분석 보고서

본 문서는 데이터베이스 이중화 및 고가용성(HA) 아키텍처 설계를 수립하기 위해, 동기식 멀티 마스터 클러스터 솔루션인 **MariaDB Galera Cluster**와 비동기식 고유 트랜잭션 식별 기술인 **GTID(Global Transaction Identifier) 기반 복제**의 작동 메커니즘, 상세 장단점, 핵심 지표 대조, 그리고 상황별 추천 설계 시나리오를 심층 비교 분석한 기술 가이드북입니다.

---

## 1. 개요 및 핵심 정체성 (Core Identity)

### 1.1 MariaDB Galera Cluster
* **정체성**: **동기식(Synchronous) Active-Active 멀티 마스터 클러스터 체제**
* **개념**: 여러 대의 DB 노드가 네트워크 상에서 하나의 단일 가상 데이터베이스 엔진처럼 동작합니다. 어느 노드에 데이터를 입력하더라도 모든 노드에 완벽하게 동시 동기화가 보장되는 철통 보안 정합성 아키텍처입니다.

### 1.2 GTID (Global Transaction Identifier) 기반 복제
* **정체성**: **비동기식(Asynchronous) Active-Passive 단방향 복제 체제 (식별 기술)**
* **개념**: 기존의 비동기 마스터-슬레이브(Master-Slave) 복제 구조를 유지하되, 모든 트랜잭션 로그에 고유 주민등록번호(UUID + Seq)인 **GTID**를 마킹하는 정밀 제어 기법입니다. 장애 복구 시 소스 코드 수준에서 유연하게 복제 방향을 역전시키고 누락 데이터를 정밀 스캔하여 복구할 수 있게 돕습니다.

---

## 2. 작동 메커니즘 및 복제 흐름 대조

### 2.1 Galera Cluster의 쓰기/동기화 흐름 (동기식)
```
[WAS] ──① Write/Commit 요청──> [DB Node 1] ──② 쓰기 인증(Certification) 검증──> [DB Node 2, 3]
  ▲                                                                              │
  └─────────────④ 성공 응답 (Commit 완료) ◄─────③ 합의 완료 (Quorum 통과) ───────┘
```
* **동기식 트랜잭션**: 쓰기가 완벽하게 클러스터 전체에 분산 복사 완료될 때까지 쓰기 세션이 대기(Lock)하므로 데이터의 100% 동시성과 정합성이 물리적으로 보장됩니다.

### 2.2 GTID 복제의 쓰기/동기화 흐름 (비동기식)
```
[WAS] ──① Write/Commit 요청──> [DB Master (GTID:45 생성)] ──② 로컬 디스크 즉시 저장 (Commit 완료) ──> [WAS 성공 응답]
                                           │ (백그라운드 비동기 배달)
                                           └─────③ GTID:45 로그 발송─────> [DB Slave (순차적 적용)]
```
* **비동기식 트랜잭션**: 마스터 자체 디스크에 저장 완료되면 WAS에 즉시 응답을 반환하여 쓰기 성능이 극도로 빠르고, 복제 패킷은 백그라운드 스레드를 통해 슬레이브로 여유롭게 흘러갑니다.

## 3. 핵심 비교 메트릭 대조군

| 비교 항목 | MariaDB Galera Cluster 🌟 | GTID 기반 비동기 복제 🔗 |
| :--- | :--- | :--- |
| **복제 지연 (Lag)** | **0초** (물리적 지연 원천 불가능) | 네트워크 부하 및 디스크 속도에 따라 미세 지연 존재 |
| **쓰기 성능 (속도)** | 모든 노드 승인을 대기하므로 상대적으로 느림 | 로컬에만 쓰고 응답하므로 **극도로 빠름** |
| **데이터 정합성** | **100% 철통 보장** (낙관적 락으로 자동 제어) | 동시 수정(UPDATE) 시 충돌/꼬임 리스크 존재 |
| **최소 노드 수** | **최소 3개 노드 필수** (Quorum 다수결 원칙) | **최소 2개 노드**만으로도 완벽 구성 가능 |
| **장애 조치 (Failover)** | **100% 엔진 수준 무인 자동** | **외부 HA 데몬 필수** (Keepalived, MHA 등) |
| **장애 복구 (Failback)** | **100% 무인 자동** (SST/IST 자동 복구) | **GTID 대조를 통한 반자동/자동** 복구 |
| **설정 복잡도** | 인프라 구조상 초기 포트 방화벽 세팅 필요 | 기존 설정에 GTID 파라미터만 켜면 되므로 비교적 간편 |

## 4. 아키텍처별 장단점 심층 분석

### 4.1 MariaDB Galera Cluster

#### 👍 주요 장점
1. **완벽한 데이터 정합성 (데이터 유실 0%)**: 
   마스터가 불시에 대파되더라도 다른 노드들이 100% 동일한 최신본을 쥐고 있으므로 데이터 실종 사고가 원천 차단됩니다.
2. **동반 자동 복구 (SST/IST)**:
   죽었던 노드가 부팅되면 클러스터가 누락 분량을 감지해 전체/증분 전송을 통해 스스로 알아서 동기화(Failback)를 완벽히 끝마칩니다.
3. **편리한 로드 밸런싱**:
   모든 노드에 읽기/쓰기가 열려 있으므로(Active-Active), 별도의 SQL 분기 튜닝 없이도 트래픽 분산이 자유롭습니다.

#### 👎 주요 단점
1. **쓰기 병목 (가장 느린 노드 성능 수렴)**:
   노드 중 단 한 장비의 디스크나 네트워크가 조금만 버벅거려도 클러스터 전체의 쓰기 속도가 그 느린 장비에 맞춰 하향 정렬됩니다.
2. **Quorum(과반수) 제약에 의한 동반 자살 위험**:
   2대로만 구성했다가 네트워크가 단절되면 두 서버가 모두 과반수를 얻지 못해(1/2 = 50%) 데이터 오염을 막기 위해 스스로 전체 서비스를 잠그고 사망(Non-Primary)합니다. (※ 해결책: 2 DB + 1 WAS(Garbd 중재자) 3노드 편법 구성 필수)

---

### 4.2 GTID 기반 비동기 복제

#### 👍 주요 장점
1. **타의 추종을 불허하는 초고속 쓰기 성능**:
   동기화 대기가 없으므로 쓰기 트랜잭션이 초당 수천 건 이상 격하게 몰리는 대용량 서비스 환경에서도 병목 없이 부드럽게 감당합니다.
2. **가볍고 유연한 인프라 구성**:
   단 2대의 서버만으로도 아무런 부담 없이 완벽하고 안정적인 이중화 망을 굴릴 수 있습니다.
3. **고유 식별자를 통한 기가 막힌 데이터 원복**:
   마스터가 죽어있는 동안 슬레이브에만 임시로 쌓였던 신규 트랜잭션을 GTID 주민번호 대조를 통해 나중에 살아난 마스터가 안전하게 쏙 골라내어 복구(Auto-Failback)할 수 있습니다.

#### 👎 주요 단점
1. **미세한 데이터 유실 위험**:
   마스터에 데이터가 저장된 순간 불시 전원 차단 등으로 대파되면, 슬레이브로 미처 전송되지 못한 0.01초 사이의 트랜잭션이 공중 유실될 위험이 존재합니다. (반동기식 복제로 일부 보완 가능)
2. **수동 복구 및 미들웨어 의존성**:
   자동 페일오버를 위해 Keepalived나 MHA, Orchestrator 같은 제3의 중재 솔루션을 연동하여 정밀 튜닝해야 하므로 인프라 관리 요소가 늘어납니다.

## 5. 최종 의사결정 추천 시나리오 (Recommendation Guide)

### 💡 시나리오 A: "Galera Cluster" 도입을 적극 권장하는 환경
1. **데이터 유실 0%가 비즈니스의 생명줄인 경우**:
   * 예: 은행, 결제 솔루션, 주문 결제 관리, 재고 세어보기 등 단 1건의 트랜잭션 누락이나 어긋남이 법적/재무적 대참사를 부르는 금융/이커머스 영역.
2. **인프라 무인화(무인 자동화)가 최우선인 경우**:
   * 야간 장애 발생 시 엔지니어가 개입하지 않고도 시스템이 알아서 죽고 살며 데이터 동기화까지 기계가 100% 무중단 복구해 내길 바라는 환경.
3. **하드웨어 가용 장비가 여유로운 경우**:
   * 최소 3대의 독립 인프라 서버를 할당할 수 있거나, WAS 서버 한구석에 가벼운 중재자(`Garbd`) 데몬을 얹어 3표 의결 구조를 완성할 여력이 있는 경우.

### 💡 시나리오 B: "GTID 비동기 복제" 도입을 적극 권장하는 환경
1. **극단적인 쓰기 속도와 대용량 트래픽 감당이 최우선인 경우**:
   * 예: 게임 로그 수집, IoT 실시간 센서 데이터 누적, 소셜 미디어 피드 업로드 등 데이터 동기화 대기가 걸리면 병목으로 서비스가 뻗어버릴 수 있는 초고속 트래픽 환경.
2. **단 2대의 DB 장비로 콤팩트하게 운영해야 하는 경우**:
   * 추가적인 중재자 프로세스나 VM 리소스를 낭비하지 않고, 가장 표준적이고 콤팩트한 2대 단독 하드웨어 구조로 탄탄하게 이중화를 굴리고 싶은 환경.
3. **통제 가능한 안전 원복(수동 개입)을 중시하는 경우**:
   * 복귀(Failback)만큼은 기계에 맡기지 않고, 정합성 대조 검증을 거친 후 엔지니어 조작 하에 완벽하게 수동/반자동(MHA 스크립트 실행)으로 마무리를 지어 안전을 다지는 정책을 쓰는 보수적인 중견/대기업 환경.
